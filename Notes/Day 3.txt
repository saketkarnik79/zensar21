Secure Coding & Compliance with Copilot
	• Secure coding principles in .NET
		Secure coding ensuresyour application resists attacks & protects data.
		Core principles of secure coding:
			Least privilege
			Defense in Depth
				Apply multiple layers:
					Input Validation
					Authentication & Authorization
					Logging & Monitoring
			Fail securely
				Do not leak implementation details
					try
					{
						// logic
					}
					catch (Exception ex)
					{
						// BAD: exposes internals
						return ex.ToString();

						// GOOD:
						return "An error occurred.";
					}
			Avoid hardcoding secrets
				// BAD
				string apiKey = "12345";

				// GOOD
				var apiKey = Configuration["ApiKey"];
			Use built-in security features
	• OWASP(Open Worldwide Application Security Project) Top 10 awareness
		OWASP is a global non-profit foundation dedicated to improving software security through open-source projects, education, and community collaboration. The most standards published by OWASP are known as "OWASP Top 10".
		Key risks & .NET Context
			Injection
				// BAD
				var query = "SELECT * FROM Users WHERE Name = '" + name + "'"; // name="xyz';DROP DATABASE AppDB;--

				// GOOD
				var cmd = new SqlCommand("SELECT * FROM Users WHERE Name = @name", conn);
				cmd.Parameters.AddWithValue("@name", name);
			Broken Authentication
			Sesitive Data Exposure
				Avoid:
					Unencypted data
					Improper storage
				Use:
					TLS
					Encryption APIs
					Secure cookies
			Security Misconfiguration
				Avoid:
					Debug mode enabled
					Open ports
				Fix:
					Harden production configs
					Disable detailed errors
			Cross-Site Scripting(XSS)
				<!-- BAD -->
				<div>@userInput</div>

				<!-- GOOD -->
				<div>@Html.Encode(userInput)</div>
			Broken Access Control

	• Optimizing LINQ & EF Core queries
		Target:
			Make database queries efficient, reduce database round trips & avoid unnecessary data loading.
				Common LINQ Issues:
					Premature Matrialization
						var users = db.Users.ToList().Where(u => u.IsActive); // Bad
						var users = db.Users.Where(u => u.IsActive).ToList(); // Good
					Selecting entire entities
						var users = db.Users.ToList(); // Bad
						var users = db.Users
										.Select(u => new { u.Id, u.Name })
										.ToList();
					N + 1 Query Issue
					Inefficient Filters
						.Where(u => u.Name.ToLower() == name.ToLower()) // Bad
						.Where(u => u.Name == name) // Good
		EF Core Optimization Techniques
			Use AsNoTracking() for read-only queries
				var users = db.Users.AsNoTracking().ToList();
			Use Compiled Queries
			Index frequently queried columns
			Paginate results
				.Skip(page * size).Take(size)

Engineering Best Practices & Team Guidelines
	• When to trust vs challenge Copilot
		Copilot is a productivity enhancer/accelerator- not a source of truth.
		When you can trust Copilot?
			Ideally, you can confidently use copilot for:
				Boilerplate code
					Controllers, DTOs, Entities, CRUD, EF Core code
				Standard patterns
					DI
					Repository Pattern
					UoW Patterns
					GoF Patterns
					SOLID Principles
				Well-known APIs
					LINQ operations
					ASP.NET Core Middleware
		When to challenge Copilot?
			Security-sensitive code
				Auth
				Encryption
				Token Validation
			Complex Logic
				Business rules
				Domain Logic
				Financial Calculations
			Performance-critical paths
				EF Core Queries
				High-Throughput APIs
			New or evolving frameworks
				Copilot may suggest outdated patterns
	In place of accepting Any suggestions blindly:
		Ask:
			Is this the best approach?
			What are the edge cases here?
			Does this follow .NET best practices?
